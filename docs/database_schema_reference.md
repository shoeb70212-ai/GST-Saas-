# Supabase Database Schema Reference

This document maps the heavily normalized, multi-tenant Postgres schema used by KhataLens to ensure strict data isolation via Row-Level Security (RLS).

## 1. Core Authentication & Tenancy

### `profiles`
Extends the native `auth.users` table to store CA-specific data.
- `id` (UUID, PK, references auth.users)
- `full_name` (Text)
- `company_name` (Text)
- `credits` (Integer) - Wallet balance for API usage
- `tier` (Text) - 'starter', 'pro', or 'custom'

### `clients`
The individual businesses managed by the CA. Acts as the primary sharding key for all RLS policies.
- `id` (UUID, PK)
- `user_id` (UUID, references profiles) - The CA who owns this client
- `client_name` (Text)
- `gstin` (Text)
- `pan` (Text)
- *RLS Policy:* `auth.uid() = user_id`

## 2. Document & Ledger Tables

### `invoices`
Stores the header-level data extracted from purchase/sales PDFs.
- `id` (UUID, PK)
- `user_id` (UUID) - Security fallback
- `client_id` (UUID, FK to clients) - Primary access filter
- `supplier_name`, `supplier_gstin`, `invoice_number`, `invoice_date`
- `total_amount`, `total_cgst`, `total_sgst`, `total_igst`
- `status` (Text) - e.g., 'scanned', 'reconciled'

### `bank_statements`
Represents a physical/digital PDF statement uploaded for processing.
- `id` (UUID, PK)
- `client_id` (UUID, FK)
- `bank_name`, `account_number`, `statement_period`
- `status` (Text) - 'processing', 'completed', 'failed'

### `bank_transactions`
The individual line items extracted from the `bank_statements`.
- `id` (UUID, PK)
- `statement_id` (UUID, FK)
- `client_id` (UUID, FK)
- `txn_date` (Date), `description` (Text)
- `withdrawal` (Numeric), `deposit` (Numeric), `balance` (Numeric)
- `status` (Text) - 'unreconciled', 'matched'

## 3. The Orchestration Bridge

### `reconciliation_suggestions`
The critical table enforcing the Human-in-the-Loop workflow. Links invoices to bank transactions.
- `id` (UUID, PK)
- `client_id` (UUID, FK)
- `invoice_id` (UUID, FK to invoices, nullable)
- `bank_transaction_id` (UUID, FK to bank_transactions)
- `match_type` (Text) - 'EXACT', 'AI_FUZZY', 'MANUAL'
- `status` (Text) - 'SUGGESTED', 'APPROVED', 'REJECTED'
- `allocated_amount` (Numeric) - For partial payments
- `confidence_score` (Numeric) - Generated by Tier 2 AI

## Supabase Functions (RPCs)
- `approve_reconciliation_match`: A PL/pgSQL function that transactionally changes the suggestion status to 'APPROVED', updates the `bank_transaction` status to 'matched', and logs the action.
- `undo_reconciliation_match`: Reverses the approval, freeing up the invoice and bank transaction for matching again.
